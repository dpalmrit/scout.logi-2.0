# PitchScout Lab — Design Spec
**Date:** 2026-05-21  
**Repo:** `scout.logi-2.0`  
**Status:** Approved

---

## Overview

A private, login-gated CV analytics lab added to the scout.logi-2.0 Next.js app. Three authorised users upload football match videos and receive full tactical analysis: radar view, Voronoi territory, ball trajectory, and match stats — all generated by a self-contained computer-vision pipeline (YOLOv8 + SigLIP + homography) running on Modal.com serverless GPU.

This is batch processing only (not real-time). A 90-min match takes 2–4 hours to process on a free T4 GPU.

## Data Model

### Job record (stored as JSON in S3 at `jobs/{jobId}/meta.json`)

```ts
interface Job {
  id: string            // nanoid
  userId: string        // email
  filename: string
  videoKey: string      // S3 key for uploaded video
  status: 'queued' | 'processing' | 'done' | 'error'
  progress: number      // 0–100
  createdAt: string     // ISO
  completedAt?: string
  error?: string
  resultsKey?: string   // S3 key for results.json when done
}
```

Jobs list on dashboard is fetched from `GET /api/jobs` — lists all `jobs/*/meta.json` for the authenticated user.

### Results JSON (written by Modal pipeline to S3 at `jobs/{jobId}/results.json`)

```ts
interface Results {
  meta: {
    fps: number
    duration: number    // seconds
    frameCount: number
  }
  frames: Array<{
    t: number           // timestamp in seconds
    players: Array<{
      id: number        // ByteTrack ID
      team: 0 | 1
      role: 'player' | 'goalkeeper' | 'referee'
      pitch_x: number   // metres on real pitch
      pitch_y: number
    }>
    ball: { pitch_x: number; pitch_y: number } | null
  }>
  trajectory: Array<[number, number]>   // [pitch_x, pitch_y] per detected frame
  stats: {
    possession: [number, number]        // [team0 %, team1 %]
    distances: Array<{ id: number; team: 0 | 1; metres: number }>
    territory: [number, number]         // avg Voronoi share [team0 %, team1 %]
  }
}
```

## Compute Pipeline (Modal)

**Location:** `modal_pipeline/pipeline.py` in the repo root  
**Deploy:** `modal deploy modal_pipeline/pipeline.py`  
**Runtime:** Python 3.11, T4 GPU, timeout 3 hours

```python
@app.function(
    gpu="T4",
    timeout=10800,
    secrets=[modal.Secret.from_name("aws-credentials")]
)
def run_pipeline(job_id: str, video_s3_key: str, results_bucket: str):
    ...
```

**Pipeline steps** (from the YouTube tutorial, order preserved):
1. YOLOv8 object detection — players, goalkeepers, referees, ball (`1280px` input resolution)
2. ByteTrack — persistent player IDs across frames
3. SigLIP + UMAP + KMeans — team classification (sample every 30 frames for training, then classify all)
4. YOLOv8 keypoint detection — 32 pitch landmarks
5. Homography (`cv2.findHomography`) — perspective transform frame→pitch
6. Per-frame: transform all player/ball positions to pitch coordinates
7. Voronoi territory — `supervision.draw_pitch_voronoi_diagram()` per sampled frame, record team territory % per frame
8. Ball trajectory — collect pitch-space positions, apply outlier filter (5m max inter-frame movement)
9. Aggregate stats — possession (frames ball nearest to team), distance per player
10. Write `results.json` to S3, update `meta.json` status → `done`

**Progress updates:** Modal writes progress directly to `jobs/{jobId}/meta.json` in S3 at checkpoints (every 10% of frames processed). The Vercel `/api/jobs/[id]/status` route reads this file — no HTTP callback from Modal required.

**Models used (Roboflow Universe, public):**
- Object detection: `football-player-detection` (YOLOv8, fine-tuned on Bundesliga data)
- Keypoint detection: `football-field-detection` (YOLOv8-pose, 32 landmarks)
- SigLIP: `google/siglip-base-patch16-224` from HuggingFace (loaded at runtime)

## Frontend Visualizations

All rendered client-side from `results.json`. No backend involvement at display time.

### Radar View
- **Canvas 2D** — pitch outline drawn as SVG background, players as coloured dots redrawn each frame
- Timeline scrubber (HTML range input) seeks to frame by index
- Play/pause button animates at `results.meta.fps` using `requestAnimationFrame`
- Team 0 = `#00e676` (green), Team 1 = `#ff5252` (red), ball = `#ffffff`

### Voronoi Territory
- **D3.js** `d3.Delaunay.from()` on player positions per frame
- SVG overlay on pitch, coloured regions, redraws on scrubber change
- Territory % shown as a bar below the pitch

### Ball Trajectory
- **SVG `<polyline>`** — all `trajectory` points drawn as a path on the pitch
- Static (full-match path), colour gradient from start (dim) to end (bright)
- Clickable: click any point on path → scrubber jumps to nearest frame

### Stats Dashboard
- Four stat cards: Possession %, Avg Distance Covered, Total Ball Touches, Territory Control
- One possession bar (two-colour horizontal bar, team0 vs team1)
- Player distance table (sorted by distance desc, shows team colour dot)

## TwelveLabs Relationship

This pipeline **augments** TwelveLabs rather than replacing it:

| Capability | CV Pipeline | TwelveLabs |
|---|---|---|
| Exact player XY coordinates | ✓ | ✗ |
| Distance covered per player | ✓ | ✗ |
| Voronoi territory control | ✓ | ✗ |
| Ball trajectory on real pitch | ✓ | ✗ |
| Persistent player IDs | ✓ | ✗ |
| Cost per match | ~$0 | $$$/min |
| Natural language querying | ✗ | ✓ |
| Semantic event detection (fouls, set pieces) | ✗ | ✓ |
| Works without model training | ✗ | ✓ |

**Recommended hybrid:** CV pipeline generates spatial/quantitative data (stats, heatmaps, positions). TwelveLabs generates semantic/event data (highlight detection, action classification). Combined: richer analysis than either alone, lower TwelveLabs bill (don't need TL for stats).

## Out of Scope (this prototype)

- Real-time / live match analysis
- Per-player heatmaps (future: easy to add from frame data)
- Formation detection
- Integration with PitchScout parent app / coach portal
- Email notifications (3 users → dashboard polling is sufficient)
- Admin UI for managing allowed users
